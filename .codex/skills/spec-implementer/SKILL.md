# Spec Implementer Skill

**Шаг ② workflow** — реализация этапов ТЗ с учётом архитектуры проекта.

## Workflow Contract

```yaml
entry:
  branch: NOT main | master
  artifacts:
    - .ai/specs/{branch-name}.md  # должен существовать
  condition: есть этапы со статусом ⬜ или 🔄

exit:
  condition: все этапы ✅
  tests: pytest passed

recommended_next_skill: test-writer  # Рекомендуемый следующий skill для Codex

uses:
  - commit-helper  # для каждого коммита этапа
```

## Предварительные требования (ENTRY проверка)

- Текущая ветка — НЕ `main`/`master`
- Существует файл `.ai/specs/{branch-name}.md`
- Есть незавершённые этапы (⬜ или 🔄)

## Алгоритм работы

### Фаза 1: Инициализация

1. **Проверить ветку:** `git branch --show-current`
   - Если main/master → STOP, предложить переключиться

2. **Найти spec:** `.ai/specs/{branch-name}.md`
   - `feature/user-auth` → `.ai/specs/feature-user-auth.md`

3. **Загрузить контекст:**
   - `.ai/ARCHITECTURE.md` — структура, Signal-Slot
   - `.ai/UI_ARCHITECTURE.md` — виджеты, компоненты

4. **Определить текущий этап:** первый с ⬜ или 🔄
   - Если все ✅ → STOP, предложить merge

---

### Фаза 2: Сбор информации

> **Цель:** собрать всю необходимую информацию перед реализацией через субагентов-трассировщиков, чтобы не делать предположений о коде.
> Детальные промпты и примеры — в [tracing-guide.md](references/tracing-guide.md).

#### Стадия 1 (обязательная): Параллельная трассировка

Запустить **два explorer sub-agent параллельно** (тип `explorer`):

- **Code Mapper** — читает все файлы этапа: классы, методы, `pyqtSignal`, `@pyqtSlot`, импорты, места использования в кодовой базе.
- **Data Flow Tracer** — трассирует request/response сигналы через `base_signals.py`, раскрывает структуры dict (все ключи и типы), находит аналогичный компонент-образец.

#### Проверка достаточности

Трассировка **достаточна** если выполнены **все** пять критериев:

| # | Критерий |
|---|----------|
| 1 | Все целевые файлы прочитаны (содержимое, не только пути) |
| 2 | Все `pyqtSignal` / `connect()` затронутых компонентов идентифицированы |
| 3 | Структуры данных request/response полностью известны (все dict-ключи и типы) |
| 4 | Найден хотя бы один аналогичный компонент-образец в кодовой базе |
| 5 | Архитектурный слой и паттерн (Core / GUI / Signal Bus) однозначно подтверждены |

Если **хотя бы один** критерий не выполнен → запустить Стадию 2.

#### Стадия 2 (условная): Углублённая трассировка

Запустить точечные субагенты только для закрытия незакрытых критериев:

- **Pattern Verifier** — ищет 2-3 аналога в кодовой базе, выявляет общий паттерн (если нет образца или неясен архитектурный слой)
- **Signal Graph Builder** — строит полный граф `signal → slot` для затронутых компонентов (если сигналы неполные)

Промпты субагентов и формат отчётов — в [tracing-guide.md](references/tracing-guide.md).

---

### Фаза 3: Реализация

1. Обновить статус: ⬜ → 🔄
2. Выполнить задачи согласно собранной информации и архитектуре (см. [architecture-patterns.md](references/architecture-patterns.md))
3. Соблюдать лимит: ≤250 строк на этап

### Фаза 4: Верификация

1. Запустить тесты: `uv run pytest`
2. Проверить размер: `git diff --stat`

### Фаза 5: Коммит

1. Conventional commit: `feat|fix|refactor|test|docs(scope): description`
2. Обновить spec: 🔄 → ✅, отметить задачи `[x]`
3. Добавить запись в историю изменений

## Статусы этапов

| Статус | Значение | Действие   |
| ------ | -------- | ---------- |
| ⬜      | Не начат | Выполнять  |
| 🔄      | В работе | Продолжить |
| ✅      | Завершён | Пропустить |

## Рекомендации следующего шага

| Ситуация       | Рекомендация                                          |
| -------------- | ----------------------------------------------------- |
| Есть ещё этапы | Продолжить следующим сообщением: "реализуй ТЗ" |
| Все этапы ✅    | Последовательно вызвать skills `test-writer`, `gui-testing`, `merge-helper` |
