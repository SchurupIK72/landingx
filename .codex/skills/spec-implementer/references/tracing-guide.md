## Стадия 1: Промпты субагентов

Оба sub-agent запускаются **параллельно** с типом `explorer`.

### Code Mapper

```
Задача: трассировка структуры кода для этапа {N} spec {branch-name}.

1. Прочитай .ai/specs/{branch-name}.md, найди этап {N}.
   Извлеки: список файлов для создания/изменения, список классов/функций.

2. Для каждого СУЩЕСТВУЮЩЕГО файла из списка:
   a. Прочитай файл полностью.
   b. Запиши: имя класса, базовые классы (наследование).
   c. Запиши все pyqtSignal с типами аргументов.
   d. Запиши все @pyqtSlot — имя метода, принимаемые типы.
   e. Запиши все публичные методы — сигнатура + краткое описание по коду.
   f. Grep по имени класса в src/ — найди все места использования.

3. Для каждого базового класса из импортов — прочитай его определение.

4. Для файлов, которые нужно СОЗДАТЬ: найди аналогичный существующий файл
   того же типа (sub_sidebar panel, core module, etc.), прочитай его как образец.

Верни структурированный отчёт:

## Файл: {path} [изменить/создать]
### Класс: {ClassName}({BaseClass})
Сигналы:
  - {signal_name} = pyqtSignal({types})
Слоты:
  - {method_name}({args}) — {описание}
Методы:
  - {method_name}({args}) — {описание}
Используется в: [{file1}, {file2}]
```

### Data Flow Tracer

```
Задача: трассировка data flow для этапа {N} spec {branch-name}.

1. Прочитай .ai/specs/{branch-name}.md, этап {N}.
   Извлеки: названия компонентов, предполагаемые action-строки.

2. Прочитай src/core/base_signals.py полностью.

3. Для каждого предполагаемого action (или компонента):
   a. Grep по "{action}" или "{ComponentName}" в src/ с контекстом (-C 3).
   b. В найденных файлах:
      - Найди emit({...}) — запиши все ключи и значения dict.
      - Найди обработчик (process_request / on_response) — запиши ключи dict.
   c. Составь полную цепочку: кто эмитирует → кто принимает.

4. Найди существующий sub_sidebar panel или core module аналогичного назначения.
   Прочитай его как образец реализации.

Верни:

## Action: "{action_name}"
- Эмитируется в: {file}:{line}
- Request dict: {"key": type, "key2": type}
- Обрабатывается в: {file}:{method}
- Response dict: {"key": type}
- Принимается в: {file}:{method}

## Образец реализации: {path}
Паттерн: {краткое описание структуры}
```

## Критерии достаточности (детально)

| # | Критерий | Достаточно | Недостаточно → запустить |
|---|----------|------------|--------------------------|
| 1 | Все целевые файлы прочитаны | Содержимое каждого файла известно | Файл не найден или не прочитан → **Pattern Verifier** |
| 2 | Сигналы/слоты идентифицированы | Все `connect()` для затронутых компонентов найдены | Есть `connect()` без известного receiver → **Signal Graph Builder** |
| 3 | Data model известна | Все dict-ключи request/response и их типы определены | Есть предположения о ключах → **Data Flow Tracer** (повтор) |
| 4 | Образец найден | Прочитан файл с аналогичной структурой в кодовой базе | Нет реального примера → **Pattern Verifier** |
| 5 | Слой подтверждён | Понятно: Core/GUI/Signal Bus + паттерн слоя | Неясно, куда поместить новый код → **Pattern Verifier** |

**Правило:** если 2 и более критериев не выполнены — запустить оба субагента Стадии 2 параллельно.

## Стадия 2: Промпты субагентов

### Pattern Verifier

```
Задача: поиск аналогичных реализаций для этапа {N} spec {branch-name}.

1. Прочитай .ai/specs/{branch-name}.md, этап {N}.
   Определи тип компонента (sub_sidebar panel / core module / plot component / etc.).

2. Найди в src/ все файлы того же типа:
   - sub_sidebar panel → src/gui/main_tab/sub_sidebar/*/
   - core module → src/core/*.py
   - plot component → src/gui/main_tab/plot_canvas/

3. Выбери 2-3 наиболее близких по назначению. Прочитай каждый полностью.

4. Выдели общий паттерн:
   - Структура __init__(self, base_signals, ...)
   - Как регистрируются сигналы
   - Как структурированы методы обработки
   - Как обновляется UI

Верни:

## Найденные аналоги
- {path1}: {назначение}
- {path2}: {назначение}

## Общий паттерн класса {тип}
```python
# Структура
```

## Специфика для данного этапа
{что отличается / на что обратить внимание}
```

---

### Signal Graph Builder

```
Задача: построить граф сигналов для компонентов этапа {N} spec {branch-name}.

1. Прочитай .ai/specs/{branch-name}.md, этап {N}. Извлеки имена компонентов.

2. Прочитай src/core/base_signals.py — запиши все определённые сигналы.

3. Для каждого компонента: grep по имени класса в src/ с контекстом (-C 5).
   В каждом найденном месте запиши все .connect() вызовы.

4. Прочитай src/gui/main_tab/main_tab.py — найди все connect() для регистрации.

Верни граф:

## Граф сигналов

{ComponentA}.{signal_name} → {ComponentB}.{slot_name}
  (определён в: {file}:{line})

## Незакрытые сигналы (нет receiver)
- {signal}: эмитируется в {file}, receiver не найден
```

## Формат итогового отчёта

После завершения обеих стадий составить итоговый отчёт перед переходом к Фазе 3:

```
📊 ТРАССИРОВКА ЗАВЕРШЕНА — Этап {N}: {stage_name}

Целевые файлы:
  ✓ {path} — [изменить] — {краткое описание содержимого}
  ✓ {path} — [создать]  — образец: {analog_path}

Data Flow:
  Action: "{action_name}"
    Request:  {"key": type, ...}
    Response: {"key": type, ...}
    Цепочка:  {EmitterClass} → base_signals → {HandlerClass}

Образец реализации: {path}

Незакрытые зависимости: нет / {список}

Стадия 2: не потребовалась / запущены [{субагенты}]

Готовность к реализации: ✅ Все критерии выполнены
```
