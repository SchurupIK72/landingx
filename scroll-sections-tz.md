# ТЗ: полноэкранный scroll-landing для PRONET SERVICES

## 1. Цель
Перевести текущий лендинг в формат `one screen per section`, где каждый основной блок занимает отдельный экран и переключается по скроллу как самостоятельный слайд.

## 2. Что считаем блоками
Отдельными scroll-screen сделать:
- `#hero`
- `#benefits`
- `#services`
- `#extras`
- `#trust`
- `#contact`

`footer` не выносить в отдельный экран.

Рекомендация:
- встроить footer в последний экран `#contact`, чтобы финальный экран был `CTA + footer`

## 3. Ожидаемое поведение
На desktop и tablet:
- один скролл переводит к следующему блоку
- каждый блок занимает высоту экрана
- активный блок всегда целиком помещается в viewport
- скролл `прилипает` к началу блока
- sticky header остается закрепленным сверху

На mobile:
- сохранить ту же идею, но без агрессивного принуждения, если контент длиннее экрана
- секции должны иметь минимум `100svh`, но допускать естественное увеличение по высоте
- если блок выше экрана, пользователь должен нормально проскроллить его содержимое

## 4. Визуальная логика
- Каждый блок воспринимается как отдельная сцена.
- Между блоками не должно быть ощущения длинной простыни.
- У каждого блока можно слегка варьировать фоновые glow-акценты, но без смены общей бренд-стилистики.
- Переходы между секциями должны быть чистыми, без резких разрывов.

## 5. Техническая реализация
В `styles.css`:
- добавить scroll snapping на уровне страницы:
- `html { scroll-snap-type: y mandatory; }`
- либо на основном контейнере, если потребуется более контролируемое поведение
- для основных секций задать:
- `scroll-snap-align: start`
- `scroll-snap-stop: always`
- высоту секций сделать через:
- `min-height: calc(100svh - header-height)`
- либо `min-height: 100svh` с учетом внутреннего top padding под sticky header
- добавить `scroll-padding-top` под высоту хедера
- привести вертикальные отступы секций к единой системе, чтобы блоки не прыгали по высоте
- у последнего блока `#contact` встроить footer внутрь общей высоты экрана

В `index.html`:
- при необходимости обернуть основные секции в общий контейнер типа `main.scroll-sections`
- footer логически привязать к `#contact`, а не оставлять как отдельную длинную нижнюю зону
- якорные ссылки из footer/header должны вести к snap-секциям корректно

## 6. Ограничения по контенту
- Нельзя ломать читаемость ради full-screen эффекта.
- Если контент блока не помещается в один экран, блок должен увеличиваться по высоте, а не обрезаться.
- Нельзя допускать, чтобы sticky header перекрывал заголовки секций.
- Нельзя делать горизонтальный скролл.

## 7. Особые требования
- Sticky header должен работать вместе со scroll snap без визуальных конфликтов.
- Якорные переходы `#hero`, `#services`, `#extras`, `#contact` должны попадать точно в начало нужного экрана.
- Локализация EN/VI не должна ломать высоту блоков.
- На мобильных браузерах использовать `svh/dvh`, а не только `vh`, чтобы избежать скачков из-за адресной строки.

## 8. Рекомендованные доработки визуала
- Hero: центрировать композицию как первый слайд.
- Benefits / Trust: карточки выровнять так, чтобы блок визуально занимал экран без лишней пустоты.
- Services / Extras: сохранить 2-колоночную логику на desktop, но держать секцию в рамках одного экрана.
- Contact + footer: собрать в единую финальную сцену с четким CTA сверху и footer снизу.

## 9. Критерии приемки
- На desktop каждый основной блок воспринимается как отдельный экран.
- Скролл прилипает к началу секции.
- Header остается закрепленным и не перекрывает контент.
- Footer не создает отдельный лишний экран.
- На mobile сайт остается удобным и не ломается из-за жесткого snap.
- Якорные ссылки и EN/VI локализация продолжают работать корректно.
